# Process Transport for CorpAPIClient

**Date:** 2026-03-14
**Status:** Approved

## Problem

The TypeScript CLI (`corp`) communicates with the API exclusively via HTTP `fetch()`. Users must either connect to the hosted API (`api.thecorporation.ai`) or start a local server (`corp serve`) and point `CORP_API_URL` at it. The Rust binary already supports a `call` subcommand that processes a single HTTP request in-memory via `tower::ServiceExt::oneshot()` — no TCP listener needed, ~6ms per call. The CLI should be able to use this directly.

## Design

### URL Scheme

The `api_url` config value determines which transport the client uses:

| `api_url` value | Transport | Behavior |
|---|---|---|
| `https://api.thecorporation.ai` | HTTP | Current default, unchanged |
| `http://localhost:8000` | HTTP | Local server over HTTP |
| `process://` | Process | Resolve binary via `@thecorporation/server`, spawn per request |
| `process:///path/to/api-rs` | Process | Use custom binary at given path |

### Transport Dispatch

All `CorpAPIClient` methods call `request(method, path, body?, params?)`. This method currently calls `fetch()`. The change: check the URL scheme first, dispatch to HTTP or process transport accordingly.

**Exceptions that bypass `request()`:**
- `provisionWorkspace()` — standalone function, setup-only. Will never receive a `process://` URL (it takes `apiUrl` as a direct parameter during `corp setup`, which always uses HTTP).
- `getPreviewPdfUrl()` — returns a raw URL string. Must throw an error when `apiUrl` is `process://` since there's no HTTP URL to construct.

### Process Transport

For each API call:

1. **Resolve binary path** (cached for client lifetime):
   - If `api_url` is `process:///custom/path`, use that path
   - Otherwise, resolve in order:
     1. `CORP_SERVER_BIN` environment variable
     2. `@thecorporation/server` npm package (`getBinaryPath()`)
     3. Local dev build: `services/api-rs/target/release/api-rs`
2. **Spawn child process:**
   ```
   api-rs --skip-validation call METHOD /path?query=params \
     -H "Authorization: Bearer {apiKey}" \
     -H "Content-Type: application/json" \
     -H "Accept: application/json" \
     [--stdin]
   ```
   The full path including query string is passed as the `path` argument — the Rust `call_oneshot` passes it directly to `uri()` which supports query strings.
3. **Pipe request body** to stdin (if present, with `--stdin` flag)
4. **Read stdout** as response body
5. **Parse stderr** for status code: scan for the last line matching `^HTTP \d+$` (tracing output also goes to stderr, so ignore non-matching lines)
6. **Return** a Response-like object implementing: `status`, `ok`, `statusText`, `json()`, `text()`, `headers`

`--skip-validation` is always passed for process transport (validation is a server-boot concern, not per-request). All env var checks in `init_state()` (JWT_SECRET, SECRETS_MASTER_KEY, etc.) still run regardless.

### Environment Variables

The process transport requires `JWT_SECRET`, `SECRETS_MASTER_KEY`, and `INTERNAL_WORKER_TOKEN`. These are the same variables auto-generated by `corp serve` into `.env`. The `.env` loading logic (`ensureEnvFile`, `loadEnvFile`) is extracted from `packages/cli-ts/src/commands/serve.ts` into a shared utility so both `serve` and process transport can use it.

### Error Handling

- **Binary not found:** Throw with message: "No api-rs binary found. Install @thecorporation/server or set CORP_SERVER_BIN"
- **Exit 1 with stderr status line** (normal API error): Extract HTTP status, pass response body through normal error handling path
- **Non-zero exit without status line** (binary crash / missing env vars): Throw with stderr content as the error message
- **Spawn failure:** Throw with the underlying OS error

### Constraints

- Response body is capped at 10MB by `call_oneshot` (line 453 of main.rs). Unlikely to be hit in practice.

### Files Changed

| File | Change |
|---|---|
| `packages/corp-tools/src/process-transport.ts` | **New.** Binary resolution, spawn, response parsing, .env loading. |
| `packages/corp-tools/src/api-client.ts` | Modify `request()` to dispatch to process transport when `process://`. Add error in `getPreviewPdfUrl()` for process URLs. |
| `packages/cli-ts/src/config.ts` | Add `process:` to allowed protocols in `validateApiUrl()`. |
| `packages/cli-ts/src/commands/serve.ts` | Extract `ensureEnvFile`/`loadEnvFile` into shared utility. |

### Files Not Changed

- All 50+ `CorpAPIClient` method signatures — they all go through `request()`
- `provisionWorkspace()` — setup-only, always receives HTTP URL
- Rust `call_oneshot` in `services/api-rs/src/main.rs` — already works

### Performance

Benchmarked on the release binary with `--skip-validation`:

| Endpoint | Response size | Time |
|---|---|---|
| `GET /health` | 17 bytes | 6ms |
| `GET /v1/openapi.json` | 289KB | 22ms |

True oneshot per request is fast enough. No process pooling or multiplexing needed.

### Binary Resolution Order

Same as `corp serve` (from `@thecorporation/server`):

1. `CORP_SERVER_BIN` env var (if set and file exists)
2. Platform-specific npm package `@thecorporation/server-{platform}-{arch}`
3. Local dev build `services/api-rs/target/release/api-rs` (relative to monorepo root)

Resolved path is cached for the lifetime of the `CorpAPIClient` instance.
