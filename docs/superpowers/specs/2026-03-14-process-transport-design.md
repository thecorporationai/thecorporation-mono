# Process Transport for CorpAPIClient

**Date:** 2026-03-14
**Status:** Approved

## Problem

The TypeScript CLI (`corp`) communicates with the API exclusively via HTTP `fetch()`. Users must either connect to the hosted API (`api.thecorporation.ai`) or start a local server (`corp serve`) and point `CORP_API_URL` at it. The Rust binary already supports a `call` subcommand that processes a single HTTP request in-memory via `tower::ServiceExt::oneshot()` — no TCP listener needed, ~6ms per call. The CLI should be able to use this directly.

## Design

### URL Scheme

The `api_url` config value determines which transport the client uses:

| `api_url` value | Transport | Behavior |
|---|---|---|
| `https://api.thecorporation.ai` | HTTP | Current default, unchanged |
| `http://localhost:8000` | HTTP | Local server over HTTP |
| `process://` | Process | Resolve binary via `@thecorporation/server`, spawn per request |
| `process:///path/to/api-rs` | Process | Use custom binary at given path |

### Transport Dispatch

All `CorpAPIClient` methods call `request(method, path, body?, params?)`. This method currently calls `fetch()`. The change: check the URL scheme first, dispatch to HTTP or process transport accordingly.

### Process Transport

For each API call:

1. **Resolve binary path:**
   - If `api_url` is `process:///custom/path`, use that path
   - Otherwise, resolve in order:
     1. `CORP_SERVER_BIN` environment variable
     2. `@thecorporation/server` npm package (`getBinaryPath()`)
     3. Local dev build: `services/api-rs/target/release/api-rs`
2. **Spawn child process:**
   ```
   api-rs --skip-validation call METHOD /path \
     -H "Authorization: Bearer {apiKey}" \
     -H "Content-Type: application/json" \
     [--stdin]
   ```
3. **Pipe request body** to stdin (if present, with `--stdin` flag)
4. **Read stdout** as response body
5. **Parse stderr** for `HTTP {status}` line to get status code
6. **Return** a Response-like object with status code and body

`--skip-validation` is always passed for process transport (validation is a server-boot concern, not per-request).

### Environment Variables

The process transport requires `JWT_SECRET`, `SECRETS_MASTER_KEY`, and `INTERNAL_WORKER_TOKEN`. These are the same variables auto-generated by `corp serve` into `.env`. The process transport reuses the same `.env` loading logic from `packages/cli-ts/src/commands/serve.ts`.

### Error Handling

- **Binary not found:** Throw with message: "No api-rs binary found. Install @thecorporation/server or set CORP_SERVER_BIN"
- **Non-zero exit:** Extract HTTP status from stderr, pass response body through normal error handling path
- **Spawn failure:** Throw with the underlying OS error

### Files Changed

| File | Change |
|---|---|
| `packages/corp-tools/src/process-transport.ts` | **New.** ~80 lines. Binary resolution, spawn, response parsing. |
| `packages/corp-tools/src/api-client.ts` | Modify `request()` to check URL scheme and dispatch to process transport when `process://`. |
| `packages/cli-ts/src/config.ts` | No schema change. `api_url` already accepts any string. |

### Files Not Changed

- All 50+ `CorpAPIClient` method signatures — they all go through `request()`
- Rust `call_oneshot` in `services/api-rs/src/main.rs` — already works
- Config schema — `process://` is just a valid `api_url` string value

### Performance

Benchmarked on the release binary with `--skip-validation`:

| Endpoint | Response size | Time |
|---|---|---|
| `GET /health` | 17 bytes | 6ms |
| `GET /v1/openapi.json` | 289KB | 22ms |

True oneshot per request is fast enough. No process pooling or multiplexing needed.

### Binary Resolution Order

Same as `corp serve` (from `@thecorporation/server`):

1. `CORP_SERVER_BIN` env var (if set and file exists)
2. Platform-specific npm package `@thecorporation/server-{platform}-{arch}`
3. Local dev build `services/api-rs/target/release/api-rs` (relative to monorepo root)
